项目背景

在项目中，我们需要将Summit系统中的数据迁移至Workday平台。Workday提供了标准化的EIB（Enterprise Interface Builder）模板，我们的任务是将Summit中的数据经过适当处理后，按照要求填入这些EIB模板中。此次数据迁移涉及多个业务领域，包括员工反馈（Feedback）、绩效期望（Expectation）以及定期检查（CheckIn）等核心模块。

引言

在此之前，我有幸向Wei Zhang同事学习并深入了解了PDD(Prompt Driven Development)这一创新方法论。这种全新的开发理念让我产生了浓厚的兴趣，因此我决定在本项目中进行实践应用。本文旨在详细记录我在实践过程中的心得体会与工作流程，希望能为各位同事提供有价值的参考和启发。

实践

流程

在数据迁移项目中，我们需要处理多个业务领域的数据。我选择从Expectation模块作为起点，开始进行了PDD实践。

初始尝试

最初的开发方式比较简单直接：

  1. 用自然语言描述所有需求细节
  2. 直接让大模型生成实现代码
  3. 人工检查运行结果

然而这种方式存在明显的问题：

  • 生成的代码质量不稳定，风格不统一
  • 运行效果难以达到预期要求
  • 需要频繁手动调整prompt

改进后的工作流

在遇到瓶颈后，我意识到可以让大模型不仅帮助实现代码，还可以协助构建高质量的Prompt。这一思路带来了工作流程的重大改进：

结果

再上述的工作流实践以及反复的人机交互之后，最终我终于得到了一份让我满意的prompt,内容如下所示:

## Requirements Description

This document aims to standardize the implementation of the Expectation data migration script, ensuring accurate conversion of source data to target format according to specified rules.

## Data Structure Definition

### Source Data Structure 
Based on @expectation.csv file, complete the following table information
Note: Headers are the header fields from the first row of @expectation.csv data

**Field Description:**
- Field Name: Field name (header)
- Field Type: Data type
- Description: Field business meaning and usage

| Field Name | Field Type | Description |
|------------|------------|-------------|
| twer_id | String | Employee unique identifier, corresponds to Worker ID in Workday |
| expectation_title | String | Title of the expectation/goal |
| expectation_content | String | Detailed description of expectation/goal, supports HTML format |
| category_titles | String | Category identifiers, multiple categories separated by "-" (e.g.: 1-2-3) |

### Target Data Structure
Based on the header fields from row 5 of @GoalEIB.xlsx file, complete the following table information

**Field Description:**
- Field Name: Field name (header)  
- Field Type: Data type
- Description: Field business meaning and usage

| Field Name | Field Type | Description |
|------------|------------|-------------|
| '' | String | Empty field, keep blank |
| Spreadsheet Key* | String | Spreadsheet unique identifier, incremental by employee |
| Worker* | String | Employee ID, from source data twer_id |
| Row ID* | String | Row identifier, incremental within employee goals |
| Goal | String | Goal field, keep blank |
| Goal Reference ID | String | Goal reference ID, keep blank |
| Name | String | Goal name, from expectation_title |
| Description | String | Goal description, from cleaned expectation_content |
| Relates To+ | String | Related field, keep blank |
| Goal Category+ | String | Goal category, converted through CATEGORY_MAPPING |

## Data Mapping Rules
Note: Target fields must include all header fields from @GoalEIB.xlsx file

**Mapping Description:**
- Target Field: Field name from @GoalEIB.xlsx
- Source Field: Corresponding field name from @expectation.csv
- Conversion Logic and Rules: Processing logic and business rules for converting source field to target field

| Target Field | Source Field | Conversion Logic and Rules |
|--------------|--------------|---------------------------|
| '' | - | Keep empty string |
| Spreadsheet Key* | - | Incremental number by employee, each unique employee assigned an incremental identifier |
| Worker* | twer_id | Direct mapping of employee ID |
| Row ID* | - | Incremental within employee goals, first goal is "1", second is "2", etc. |
| Goal | - | Keep empty string |
| Goal Reference ID | - | Keep empty string |
| Name | expectation_title | Direct mapping of goal title, only shown in first row of each category, empty for others |
| Description | expectation_content | Mapped after cleaning HTML tags via cleanHTML function, only shown in first row of each category, empty for others |
| Relates To+ | - | Keep empty string |
| Goal Category+ | category_titles | Map category IDs to Workday category codes via CATEGORY_MAPPING constant (1→GOAL_CATEGORY-3-1, 2→GOAL_CATEGORY-3-6, etc.) |

## Structure Design

### Method Definition
1. **migrateData** - Main function coordinating entire data migration process, calls other methods in sequence and handles exceptions
2. **readData** - Reads source data file, parses CSV content into structured data
3. **readEmployeeRegionMapping** - Reads employee region mapping file, establishes employee-region correspondence
4. **groupDataByRegion** - Groups data by region, supports special region handling (e.g., Germany, Terminated)
5. **transformData** - Converts source data to target data format according to mapping rules
6. **writeToExcel** - Writes converted data to target Excel file

### Template File Definition

| Template File | Responsibility | Template Path |
|---------------|----------------|---------------|
| GoalEIB | EIB template for Expectation | `./template/GoalEIB.xlsx` |

## Task Implementation

### Implement readData Method
- **Parameters**: filePath (source data file path)
- **Logic**:
    1. Use fs.readFileSync to read specified CSV file
    2. Use csv-parse/sync to parse CSV content into structured data
    3. Handle potential file reading errors
- **Return**: Array of parsed records (each element is an object representing one row)

### Implement readEmployeeRegionMapping Method
- **Parameters**: None (uses fixed employee2.csv file path)
- **Logic**:
    1. Read employee data file
    2. Create region mapping based on employee status and office location
    3. Special handling: Terminated employees go to "Terminated" group, Germany office employees (except leaders) go to "Germany" group
    4. Other employees grouped by region_name
- **Return**: Object mapping employee IDs to region names

### Implement groupDataByRegion Method
- **Parameters**: sourceData (original data array), regionMapping (region mapping object)
- **Logic**:
    1. Look up corresponding region in region mapping by employee ID
    2. Group data by region
    3. Count data volume by region
- **Return**: Object of data grouped by region

### Implement transformData Method
- **Parameters**: sourceData (original data array)
- **Logic**:
    1. Group data by employee
    2. Assign incremental Spreadsheet Key for each employee
    3. Assign incremental Row ID for each employee's goals
    4. Process category fields: split category_titles by "-", generate one row per category
    5. Clean HTML tags from expectation_content using cleanHTML
    6. Map category IDs to Workday category codes via CATEGORY_MAPPING
    7. For multi-category goals, first row contains complete information, subsequent rows only contain category information
- **Return**: Array of transformed data

### Implement writeToExcel Method
- **Parameters**: transformedData (processed data), outputFile (target Excel file path)
- **Logic**:
    1. Check and delete existing target file
    2. Copy template Excel file to target path
    3. Open target Excel file using ExcelJS
    4. Locate specific worksheet ("Manage Goals")
    5. Determine data start row (by finding first empty row)
    6. Establish column mapping relationship (based on row 5 headers)
    7. Write data to worksheet row by row
    8. Save Excel file
- **Return**: Promise (async operation completion)

### Implement migrateData Method
- **Parameters**: None
- **Logic**:
    1. Control entire migration process
    2. Call readData, readEmployeeRegionMapping, groupDataByRegion, transformData, and writeToExcel methods in sequence
    3. Process statistics for expectations containing table content
    4. Generate separate Excel file for each region
    5. Handle process exceptions and errors
    6. Add appropriate log outputs
- **Return**: No return value or Promise

## Special Processing Logic

### HTML Content Cleaning
- Use cleanHTML function to process expectation_content field
- Remove unsafe HTML tags and attributes
- Preserve allowed format tags (e.g., b, i, u, p, ul, ol, li)
- Validate link validity, remove invalid href attributes

### Category Mapping Processing
- Use CATEGORY_MAPPING constant for category conversion
- Mapping relationship: 1→GOAL_CATEGORY-3-1, 2→GOAL_CATEGORY-3-6, 3→GOAL_CATEGORY-3-5, 4→GOAL_CATEGORY-3-4, 5→GOAL_CATEGORY-3-3, 6→GOAL_CATEGORY-3-2
- Throw clear error message for unmapped categories

### Region Grouping Special Rules
- Terminated status employees grouped separately as "Terminated"
- Retired status employees grouped separately as "Retired"
- Germany office employees (except specific leaders) grouped separately as "Germany"
- Other employees grouped by original region_name

### Multi-Category Data Processing
- When a goal has multiple categories, expand into multiple rows by category count
- First row contains complete Name and Description information
- Subsequent rows have empty Name and Description fields, only containing different Goal Category+ values
- Worker*, Spreadsheet Key*, Row ID* remain consistent across all rows

## Constraints
- Use Node.js environment
- Ensure data conversion accuracy and completeness
- Add necessary console log outputs
- Field type conversion: Pay attention to data type conversion, ensure compliance with target field format requirements
- Required dependencies:
    - `fs`: For file system operations, reading and writing files
    - `path`: For handling file paths, providing cross-platform path tools
    - `csv-parse/sync`: For parsing CSV file content into structured data
    - `exceljs`: For handling Excel files, supporting reading, modification, and writing
    - `jsdom`: For HTML content cleaning and DOM operations

## Output File Naming Rules
- Generate separate Excel files by region
- File name format: `ExpectationEIB_{region_name}.xlsx`
- Replace "/" character in region names with "-" to ensure valid filename
- Examples: `ExpectationEIB_North America.xlsx`, `ExpectationEIB_India-Middle East.xlsx`

## Data Quality Monitoring
- Count expectations containing HTML table content
- Record invalid link handling situations
- Output data distribution statistics by region
- Verify completeness and correctness of category mapping

实践感受

效率与质量的双重提升

在应用PDD方法进行数据迁移项目的过程中，我深刻体会到这种开发模式带来的显著优势。首先，通过精心设计的prompt，我能够快速获取高质量的代码框架和实现思路，大幅缩短了从需求分析到代码实现的时间。特别是在处理Summit到Workday的数据转换逻辑时，通过与AI的多轮对话，我能够逐步完善和优化解决方案，确保代码的健壮性和可维护性。

创造力

令我惊喜的是，PDD开发模式激发了我的创造力。在与AI的协作过程中，我不再局限于已知的解决方案，而是能够探索更多可能性。例如，在处理Summit中的非标准数据格式时，通过描述问题本质并与AI讨论，我们共同设计了一套灵活的数据清洗和转换策略，这是我最初未能想到的方案。

效能提升与知识获取

PDD方法不仅提高了开发效率，还成为我学习新技术和领域知识的有力工具。在面对Workday EIB模板的复杂结构时，我通过向AI提问并分析其回答，快速掌握了数据映射的关键点和最佳实践。这种交互式学习方式比传统的文档阅读更加高效，也更符合实际工作中的问题解决流程。

解决方案着手

PDD方法最大的收获之一是我的思维模式发生了根本性转变。在传统开发中，我习惯于立即深入实现细节，关注具体的代码结构、变量命名和算法选择等微观层面的问题。而在采用PDD后，我学会了首先从解决方案的整体架构着手，关注问题的本质和解决思路。
这种从”实现细节着手”到”解决方案着手”的转变，使我能够以更高的视角审视问题，避免过早陷入技术细节的泥潭。在数据迁移项目中，我不再一开始就纠结于具体的数据处理函数如何编写，而是先理清数据流转的整体路径、关键转换节点和验证策略，然后再逐步细化到具体实现。这种自上而下的思考方式不仅提高了解决复杂问题的能力，也使最终的代码设计更加清晰合理。

抽象思维

在PDD开发模式的实践中，我发现抽象思维能力得到了显著提升。传统编程中，我们常常直接从具体实现出发，而PDD则鼓励我先描述问题和解决方案的抽象模型，再逐步细化。这种方式培养了我从更高层次思考问题的能力。

在数据迁移项目中，我不再局限于”如何编写特定的数据转换函数”这样的具体问题，而是先思考”数据转换的本质是什么”、”哪些转换模式可以抽象复用”等更为抽象的问题。通过与AI的对话，我能够更清晰地表达这些抽象概念，并将它们转化为可实现的代码结构。

特别值得一提的是，PDD与抽象思维形成了良性循环：一方面，抽象思维帮助我构建更有效的提示词，使AI能够理解问题的本质；另一方面，AI生成的解决方案又常常包含我未曾想到的抽象模式，这反过来丰富了我的抽象思维库。

提示词能力的培养与提升

在PDD开发模式中，提示词（Prompt）的设计与表达能力成为了决定开发效率和质量的关键因素。通过这个项目，我深刻认识到优质提示词的重要性，并在实践中不断提升这一能力。一个好的提示词应当清晰表达问题本质、提供必要的上下文信息、明确期望输出，同时避免过度约束AI的创造性思考空间。

在数据迁移项目初期，我的提示词往往过于粗糙和粗力度，导致AI生成的代码要么偏离需求，要么缺乏灵活性。随着经验积累，我学会了分层次构建提示词：先描述整体目标和业务背景，再明确技术约束和具体需求，最后提出对代码质量的期望。这种结构化的提示词极大提高了AI响应的准确性和实用性。

特别值得一提的是，我发现提示词能力与领域知识深度密切相关。对Summit和Workday系统理解越深入，我就能越精准地描述数据转换需求，从而获得更符合业务逻辑的代码实现。这种”提示词-领域知识”的良性循环，成为我在PDD实践中持续进步的动力源泉。

Tasking能力的提升

在PDD开发模式中，我发现自己的任务分解（Tasking）能力得到了显著提升。传统开发中，我常常面临任务拆分不够清晰或粒度不合适的问题，导致开发过程中频繁调整计划。而通过与AI的协作，我学会了如何更科学地将复杂问题分解为可管理的子任务。

在数据迁移项目中，我首先通过与AI对话，梳理出整个迁移流程的主要阶段：数据提取、清洗转换、数据注入。然后针对每个阶段，我又进一步细化为具体的功能模块和实现步骤。

特别值得一提的是，AI常常能够指出我在任务分解中忽略的环节，如数据一致性检查、异常处理策略等。这些建议不仅完善了当前项目的规划，也丰富了我对软件开发全生命周期的理解。通过不断实践，我逐渐形成了”先全局后局部、先框架后细节”的任务分解思维模式，大大提高了工作效率和代码质量。

挑战与应对

当然，PDD方法也有其挑战。如何准确表达需求、如何评估AI生成代码的质量、如何在保持效率的同时确保对核心业务逻辑的理解，这些都是我在实践中不断思考和解决的问题。通过建立清晰的prompt模板、设计验证机制以及保持对关键业务知识的学习，我逐步形成了一套适合自己的PDD工作流程。
