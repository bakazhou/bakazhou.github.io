Workday Integration Guide

Background

Due to project requirements, I recently started learning and exploring Workday system integration knowledge. Prior to this, my understanding of Workday was limited to knowing it as one of the world’s leading cloud-based HCM solution providers, and having uploaded some required personal information during onboarding - overall, I had almost no knowledge of this system. Based on the practices and understanding gained during the exploration of integration, this article records some summaries and insights, hoping to provide reference and help for colleagues with similar needs.

Overview of Workday Integration

Integration Types Overview

Workday provides multiple integration methods, which can be mainly divided into the following categories:

1. Inbound Integration

Importing external data into Workday, such as batch importing employee data and updating organizational structures. Mainly implemented through EIB or Web Service, requiring strict data validation.

2. Outbound Integration

Exporting data from Workday to external systems, commonly used for employee information synchronization, financial data export, etc. Can be implemented using RaaS (Report as a Service), supporting flexible data formats and filtering conditions.

3. Real-time Integration

Implementing real-time data exchange between systems through Web Service or REST API, suitable for scenarios like employee status synchronization and real-time approvals. Performance and system availability need to be considered.

4. Scheduled Integration

Scheduled periodic integration tasks, suitable for handling large volumes of data, such as daily data synchronization and periodic report generation. Resource consumption is controllable with lower maintenance costs. This type of integration can be implemented through external system scheduling or through Workday’s Integration System. Integration System is currently in the exploration phase.

5. Event-driven Integration

Integration triggered by specific business events, such as employee onboarding/offboarding processing and organizational change notifications. Supports complex business processes and requires event subscription configuration. Like scheduled integration, it can be implemented through Integration System, which is currently being explored.

Summary of the content of this article

This article will focus on two integration methods: Inbound Integration and Outbound Integration. The main content includes:

  • Workday core component introduction

    • Security Group

    • Report

  • Integration task practice

    • Report as a Service Integration

核心组件

Security Group

概念

Security Group is a core component of permission management in Workday. It is a collection of users designed to manage access rights to protectable items within Workday tenants. Through Security Groups, administrators can centrally manage and distribute user permissions to achieve precise control over system resources.

Functions and Purposes

Security Groups provide the following key functionalities in the Workday system:

  1. Centralized Permission Management

    • Unified management of user and role permissions
    • Support for bulk permission assignment and revocation
    • Can include users, roles and other security groups

  2. Access Control

    • Control access permissions for business objects
    • Manage feature and operation permissions
    • Restrict data viewing and modification scope

  3. System Integration Authorization

    • Provide necessary access permissions for integration systems
    • Control interface call permissions for external systems
    • Manage access scope for APIs and Web Services

Integration System User

Concept

Integration System User (ISU) is a special user type in Workday specifically designed for system integration. Unlike regular users, ISU is mainly used for automated interactions between systems rather than manual operations. ISU plays an important role in Workday’s integration architecture, serving as a bridge connecting Workday with external systems to ensure secure communication between systems.

Use Cases

ISU is mainly used in the following scenarios:

  1. Web Service integration (including but not limited to REST, SOAP, XML, etc.)
  2. Enterprise Interface Builder (EIB) import/export
  3. Report as a Service
  4. Data synchronization and processing

创建步骤

  1. Use the following Task to create a Security Group in Workday:

    Create Integration System User

  2. Fill in the basic information:

    • Username: Follow naming conventions
    • Password: Comply with security policies

After filling in the information, click the OK button.

After completing the above steps, you will have an ISU named ISU_Tianle for integration tasks.

API Client

Concept

API Client is an important component in Workday for managing and controlling API access. It serves as an interface between applications and the Workday API, handling authentication and authorization. API Client ensures API call security through mechanisms like JWT (JSON Web Token).

Use Cases

API Client is mainly used in the following scenarios:

  1. External application access to Workday API
  2. System integration development
  3. Automated process implementation
  4. Data synchronization and exchange

Creation Steps

  1. Use the following Task to create an API Client in Workday:
    Create API Client
  2. Fill in the basic information:
    • Client Name: Follow naming conventions
    • Authentication Method: Select JWT Bearer Grant
    • X509 Certificate: Upload generated public_cert.pem file (see Notes below for details)
    • Allow Integration Messages: Checked
    • Grant Admin Consent: Checked
    • Scope (Function Areas): Select required Function Areas (Tenant Non-Configurable is mandatory)

Important Notes

  1. When creating an API Client, you need to generate an x509 certificate using the following method:
    openssl req -x509 -newkey rsa:2048 -keyout private_key.pem -out public_cert.pem -days 365
openssl rsa -in private_key.pem -pubout -out public_key.pem
openssl pkcs8 -topk8 -inform PEM -outform PEM -nocrypt -in private_key.pem -out private_key_pkcs8.pem
  2. After generating the certificate, fill in the value of public_cert.pem into Workday:
    cat public_cert.pem

After filling in the information, click the OK button.

After completing the above steps, you will have an API Client named Tianle_Client for integration tasks.

Report

Overview

Report is one of the core functions in the Workday platform, providing powerful data processing and reporting capabilities. Through Reports, users can integrate scattered data into valuable datasets to provide data support for decision making. It not only supports basic data extraction functions, but can also handle complex business scenarios, including but not limited to:

  • Custom report design: Supports flexible field selection
  • Multi-data source integration: Can extract and correlate data from multiple data sources
  • Advanced calculation functions: Supports complex data calculations and statistical analysis (Calculated Fields)
  • Dynamic parameter configuration: Allows users to set dynamic filtering conditions
  • Formatted output: Supports multiple output formats such as Excel, PDF, etc.
  • Web Service integration: Can serve as a data service for other systems to call

As an indispensable data management tool in the Workday platform, Report provides enterprises with comprehensive data visualization and analysis capabilities.
Report is mainly used in the following business scenarios:

  1. Data analysis and report generation
  2. System integration data source
  3. Business monitoring and decision support
  4. Compliance report generation

Creation Steps

  1. Use the following Task to create a Report in Workday:
    Create Custom Report
  2. Fill in the basic information:
    • Report Name: Follow naming conventions
    • Report Type: Select Advanced
    • Enable Web Service: Checked
    • Optimized Performance: Checked

After filling in the information, click the OK button.

After completing the above steps, you will have a Report named Tianle_Test_Report for integration tasks.

Notes

  1. When selecting data sources, if business characteristics require real-time data, it is recommended to use non-indexed data sources.
  2. If you want to choose a non-indexed data source, you need to uncheck the Optimized for Performance option.

Web Service Integration

Concept

Web Service Integration is a standardized interface service provided by Workday. It allows external systems to conduct data interaction with Workday through the HTTP/HTTPS protocol to operate data in Workday. Workday supports multiple web service types, including:

  1. REST Web Services

    • Based on REST architecture style
    • Use JSON data format
    • Support standard HTTP methods (GET, POST, PUT, DELETE, etc.)

  2. SOAP Web Services

    • Based on SOAP protocol
    • Uses XML data format
    • Provides WSDL interface description

The following paragraph comes from the official Workday documentation:

Workday REST APIs exist alongside our public SOAP-based web services. For transferring large sets of data, use the SOAP web services. SOAP web services best support batch transactions for external enterprise applications.

The official Workday documentation mentions that REST API coexists with SOAP API, and for batch data transmission, it is recommended to use SOAP API. Moreover, in Workday, SOAP API provides richer resource access, therefore in this article, the main examples will use SOAP API.

Note: SOAP API may not be common. If you are interested, you can refer to the following document for a basic understanding: https://blog.postman.com/soap-api-definition/. We will not elaborate further here.

Prerequisites

  1. Created Integration System User (ISU_Tianle)
  2. Configured API Client (Tianle_Client)
  3. Created Security Group required for integration tasks (Tianle_ISSG)

Authentication

Workday Web Service supports the following authentication methods:

  1. Basic Authentication

    • Uses username and password
    • Suitable for simple integration scenarios
    • Relatively lower security

  2. OAuth 2.0

    • Supports JWT authorization flow
    • Provides higher security
    • Supports access token refresh

Based on the introduction of the API Client here, we choose the OAuth 2.0 authentication method.

Summary and Thoughts

Integration Key Points

Through the learning and practice of Workday integration, the following key points are worth noting:

  1. Permission Management is Very Important

    • Security Group is the core of permission management
    • Follow the principle of least privilege
    • Permission changes need to be activated promptly

  2. Choice of Authentication Method

    • OAuth 2.0 + JWT is the recommended authentication method
    • Need to properly store certificate and key information
    • Regularly update authentication information

  3. Selection of Integration Method

    • Choose appropriate integration method based on business scenarios
    • RaaS is suitable for scenarios requiring custom data queries
    • Web Service is suitable for complex business operations

Mastering the two integration methods mentioned in this article is sufficient to solve some integration problems. I hope this article can help colleagues who also need to integrate with Workday. If you have any questions, please feel free to discuss and exchange ideas.

Practical Experience

During the exploration process, we summarized the following practical experiences to share with everyone:

  • Due to the lack of clear official learning paths and documentation guidelines, the learning curve is steep. During the exploration process, you need to invest significant time and energy in trial and error, and verify through practical operations as much as possible.
  • The content of Workday’s official documentation is sometimes lacking. It is recommended to utilize AI tools to assist in searching for information, which can improve the efficiency of knowledge integration.
  • The Workday Community platform is recommended. Although the content there is relatively brief, it contains a wealth of practical experience sharing and serves as good reference material.
  • Workday integration tasks usually require multiple steps working together to complete, so patience and attention to detail are necessary during the operation.

Future Outlook

As a feature-rich and massive system, truly mastering Workday’s integration capabilities is no easy task. While self-study and practice can solve some basic problems, systematic training and coursework are essential for gaining an in-depth understanding of this field.
