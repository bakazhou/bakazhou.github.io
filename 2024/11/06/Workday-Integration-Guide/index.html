background

Due to project demand, we have recently started learning and exploring the integrated knowledge of the Workday system. Based on the practice and understanding of this process, this article records some summaries and experiences, hoping to provide reference and help for colleagues with similar needs.

Workday Integrated Overview

Integrated Type Overall

Workday provides a variety of integration methods, which can be divided into the following categories:

1. Inbound Integration (data import)

Import external data into WorkDay, such as importing employee data and updating the organizational structure. Mainly implemented through EIB or Web Service, and strict data verification is required.

2. OutBound Integration (data export)

Export data from WorkDay to external systems are often used for employee information synchronization and financial data export. It can be implemented using RaaS (Report as a Service) to support flexible data formats and filtering conditions.

3. Real-Time Integration (real-time integration)

Real -time data exchange between systems is implemented through Web Service or REST API, which is suitable for employee status synchronization and real -time approval and other scenarios. Performance and system availability need to be considered.

4. Scheduled Integration (timing integration)

The regular integration tasks performed according to the plan are suitable for processing a large amount of data, such as daily data synchronization and regular report generation. Resource consumption is controllable, and the maintenance cost is low. This type of integration can be implemented in timing tasks in the external system, or it can be implemented through the Integration System in Workday. Integration System is currently in the exploration stage.

5. Event-Driven Integration (event drive)

Integration triggered by specific business events, such as employees’ entry/resignation processing, organization change notice, etc. Support complex business processes, and you need to configure event subscriptions. Like timing integration, it can be implemented through the Integration System and is currently exploring the stage.

Core component

Security Group

concept

Security Group is the core component of authority management in WorkDay. It is a user collection to manage the access rights that can protect the projects in the Workday tenants. Through the security group, administrators can centrally manage and distribute user authority to achieve precise control of system resources.

Function and purpose

The security group mainly provides the following functions in the WorkDay system:

  1. Centralized Permission Management

    • Unified management of user and role permissions
    • Support batch assignment and revocation of permissions
    • Can include users, roles and other security groups

  2. Access Control

    • Control access permissions for business objects
    • Manage permissions for features and operations
    • Restrict data viewing and modification scope

  3. System Integration Authorization

    • Provide necessary access permissions for integration systems
    • Control interface call permissions for external systems
    • Manage access scope for APIs and Web Services

Integrating system user

concept

Integrating system user (ISU) is a special user type specifically for system integration. Unlike ordinary users, ISU is mainly used for automated interaction between systems, not artificial operations. ISU plays an important role in the workday integrated architecture. It is a bridge connecting Workday and external systems to ensure safe communication between systems.

Use scene

ISU is mainly used in the following scene:

  1. Web service integration (including but not limited to REST, SOAP, XML, etc.)
  2. Enterprise Interface Builder (EIB) Import/Export
  3. Report self -service
  4. Data synchronization, data processing

API Client

concept

API Client is an important component for managing and controlling API access in WorkDay. As an interface between the application and the WorkDay API, it is responsible for processing certification and authorization. API Client ensures the security of API by JWT (JSON Web token) and other mechanisms.

Use scene

API Client is mainly used in the following scenes:

  1. External applications access workday API
  2. System integration development
  3. Automatic process implementation
  4. Data synchronization and exchange

Create steps

  1. Use below Task create API Client in WorkDay:
    Create API Client
  2. Fill in the basic information:
    • Client Name: Follow naming conventions
    • Authentication Method: Select JWT Bearer Grant
    • X509 Certificate: Upload generated public_cert.pem file (see Notes below for details)
    • Allow Integration Messages: Checked
    • Grant Admin Consent: Checked
    • Include Workday Owned Scope: Checked
    • Scope(Function Areas): Select necessary Function Areas (Tenant non-configuration is a must-have)

Important Notes

  1. When creating the API Client, you need to generate an X509 certificate. The generating method is as follows:
    openssl req -x509 -newkey rsa:2048 -keyout private_key.pem -out public_cert.pem -days 365
openssl rsa -in private_key.pem -pubout -out public_key.pem
openssl pkcs8 -topk8 -inform PEM -outform PEM -nocrypt -in private_key.pem -out private_key_pkcs8.pem
  2. After generating a certificate, fill in the value of the public_cert.pem into workday.
    cat public_cert.pem

After filling in the information, click OKButton.

After completing the above steps, it has an API Client name for integrated tasks.Tianle_Client.

Report

Overview

Report is one of the core functions in the workday platform, which provides powerful data processing and reporting ability. Through Report, users can integrate decentralized data into valuable data sets to provide data support for decision -making. It not only supports basic data extraction functions, but also can handle complex business scenarios, including but not limited to:

  • Custom report design: Support flexible field selection
  • Multi -data source integration: data can be extracted and related data from multiple data sources
  • Advanced computing function: support complex data computing and statistical analysis (Calculated Fields)
  • Dynamic parameter configuration: allow users to set dynamic filtration conditions
  • Format output: Support multiple output formats, such as Excel, PDF, etc.
  • Web service integration: can be used as a data service for other system calls

As an indispensable data management tool in the WorkDay platform, REPORT provides enterprises with comprehensive data visualization and analysis capabilities.
Report is mainly used in the following business scenarios:

  1. Data analysis and report generation
  2. System integrated data source
  3. Business monitoring and decision support
  4. Compliance report generation

Create steps

  1. Use below Task create Report in WorkDay:
    Create Custom Report
  2. Fill in the basic information:
    • Report Name: Follow naming conventions
    • Report Type: Select Advanced
    • Enable Web Service: Checked
    • Optimized Performance: Checked

After filling in the information, click OKButton.

After completing the above steps, there is a report name for the integrated task Tianle_Test_Report.

  1. When the data source is selected, if the business characteristics need to be real -time data, the non -index data source is recommended.
  2. If you want to choose a non -index data source, then you need to cancel the check Optimized for PerformanceOptions.

Web service integration

concept

Web Service Integration is a standardized interface service provided by Workday. It allows external systems to conduct data interaction with WorkDay through the HTTP/HTTPS protocol to operate data in Workday. Workday supports multiple web service types, including:

  1. Rest web services

    • Based on REST architecture style
    • Use JSON data format
    • Support standard HTTP methods (GET, POST, PUT, DELETE, etc.)

  2. SOAP Web Services

    • Based on SOAP protocol
    • Uses XML data format
    • Provides WSDL interface description

The following paragraph comes from the official document of Workday:

Workday REST APIs exist alongside our public SOAP-based web services. For transferring large sets of data, use the SOAP web services. SOAP web services best support batch transactions for external enterprise applications.

The official document of Workday mentioned that REST API coexisted with SOAP API. For batch data transmission, it is recommended to use SOAP API. And in WorkDay, the resource access provided by SOAP API is more rich and rich, so in this article, the main example is SOAP API.

Notes: SOAP API may not be common. If you are interested, you can refer to the following documents, and you can simply understand: https://blog.postman.com/soap-api-definition/ here will not be repeated.

Prerequisite

  1. Created Integration System User (ISU_Tianle)
  2. API Client (Tianle_Client)
  3. Security Group (Security Group (Tianle_ISSG)

Authentication

Workday Web Service supports the following certification methods:

  1. Basic Authentication

    • Uses username and password
    • Suitable for simple integration scenarios
    • Relatively lower security

  2. OAuth 2.0

    • Supports JWT authorization flow
    • Provides higher security
    • Supports access token refresh

Based on the introduction of the API Client here, we choose the OAUTH 2.0 certification method.

Summary and thinking

Integrated main point summary

Through the learning and practice of workday integration, the following key points are worth noting:

  1. Power management is very important

    • Security Group is the core of permission management
    • Follow the principle of least privilege
    • Permission changes need to be activated promptly

  2. Choice of authentication method

    • OAuth 2.0 + JWT is the recommended authentication method
    • Need to properly store certificate and key information
    • Regularly update authentication information

  3. Selection of Integrated Method

    • Choose appropriate integration method based on business scenarios
    • RaaS is suitable for scenarios requiring custom data queries
    • Web Service is suitable for complex business operations

Master the two integration methods mentioned in this article, which is enough to solve the integrated problems of the part. I hope this article can help colleagues who also need to be integrated with workday. If you have any questions, please exchange and discuss.

Practical experience

During the exploration process, we summarized the following practical experience and shared it with everyone:

  • Due to the lack of clear official learning paths and clear document guidelines, the learning curve is steep. During the exploration process, you need to invest a lot of time and energy for trial and error, and to verify as much as possible.
  • The content of the official documentation of workday is lacking. It is recommended to use AI tools to assist in searching for data, which can improve the efficiency of knowledge integration.
  • It is recommended to use the Workday Community platform. Although the content above is relatively simple, it contains a lot of practical experience sharing, which is a good reference material.
  • Workday integration tasks usually require multiple steps to cooperate to complete, so it is necessary to maintain patience and carefulness during the operation.

Future Outlook

As a functional and huge system, Workday is not easy to really master its integration ability. Although it can solve some basic problems through self -study and practice, it is necessary to understand this field in depth, systematic training and curriculum learning must be essential.
