Workday Integration Guide

Background

Due to project requirements, I recently started learning and exploring Workday system integration knowledge. Prior to this, my understanding of Workday was limited to knowing it as one of the world’s leading cloud HCM solution providers, and having uploaded some required personal information during onboarding. Overall, I had very little knowledge of this system. Based on my practice and understanding during the integration exploration process, this article records some summaries and insights, hoping to provide reference and help for colleagues with similar needs.

Workday Integration Overview

Integration Types Overview

Workday provides multiple integration methods, which can be mainly divided into the following categories:

1. Inbound Integration

Import external data into Workday, such as batch importing employee data and updating organizational structure. Mainly implemented through EIB or Web Service, requiring strict data validation.

2. Outbound Integration

Export data from Workday to external systems, commonly used for employee information synchronization, financial data export, etc. Can be implemented using RaaS (Report as a Service), supporting flexible data formats and filtering conditions.

3. Real-time Integration

Implement real-time data exchange between systems through Web Service or REST API, suitable for employee status synchronization, real-time approval, and other scenarios. Need to consider performance and system availability.

4. Scheduled Integration

Scheduled integration tasks that execute periodically, suitable for handling large amounts of data, such as daily data synchronization, periodic report generation, etc. Resource consumption is controllable with lower maintenance costs. This type of integration can be implemented through scheduled tasks in external systems or through Integration System in Workday. Integration System is currently in the exploration phase.

5. Event-driven Integration

Integration triggered by specific business events, such as employee onboarding/offboarding processing, organizational change notifications, etc. Supports complex business processes and requires event subscription configuration. Like scheduled integration, it can be implemented through Integration System, which is currently in the exploration phase.

Core Components

Security Group

Concept

Security Group is a core component of permission management in Workday. It is a collection of users used to manage access permissions to protectable items in the Workday tenant. Through security groups, administrators can centrally manage and assign user permissions to achieve precise control over system resources.

Functions and Uses

Security groups in Workday system mainly provide the following functions:

  1. Centralized Permission Management

    • Unified management of user and role permissions
    • Support batch assignment and revocation of permissions
    • Can include users, roles, and other security groups

  2. Access Control

    • Control access permissions to business objects
    • Manage function and operation usage permissions
    • Restrict data viewing and modification scope

  3. System Integration Authorization

    • Provide necessary access permissions for integration systems
    • Control interface call permissions for external systems
    • Manage API and Web Service access scope

Creation Steps

Use the following Task in Workday to create a Security Group

Create Security Group

Integration System User

Concept

Integration System User (ISU) is a special user type in Workday specifically for system integration. Unlike regular users, ISU is mainly used for automated interaction between systems rather than manual operations. ISU plays an important role in Workday’s integration architecture, serving as a bridge connecting Workday with external systems to ensure secure communication between systems.

Use Cases

ISU is mainly applied in the following scenarios:

  1. Web Service integration (including but not limited to REST, SOAP, XML, etc.)
  2. Enterprise Interface Builder (EIB) import/export
  3. Report self-service
  4. Data synchronization and processing

Creation Steps

  1. Use the following Task in Workday to create a Security Group:

    Create Integration System User

  2. Fill in basic information:

    • Username: Follow naming conventions
    • Password: Comply with security policy

After filling in the information, click the OK button.

After completing the above steps, you will have an ISU named ISU_Tianle for integration tasks.

API Client

Concept

API Client is an important component in Workday for managing and controlling API access. It serves as an interface between applications and Workday APIs, responsible for handling authentication and authorization. API Client ensures API call security through mechanisms such as JWT (JSON Web Token).

Use Cases

API Client is mainly applied in the following scenarios:

  1. External application access to Workday API
  2. System integration development
  3. Automation process implementation
  4. Data synchronization and exchange

Creation Steps

  1. Use the following Task in Workday to create an API Client:
    Create API Client
  2. Fill in basic information:
    • Client Name: Follow naming conventions
    • Authentication Method: Select JWT Bearer Grant
    • X509 Certificate: Upload generated public_cert.pem file (see Notes below)
    • Allow Integration Messages: Check
    • Grant Admin Consent: Check
    • Scope(Function Areas): Select necessary Function Areas (Tenant Non-Configurable is mandatory)

Important Notes

  1. When creating an API Client, you need to generate an x509 certificate, generation method as follows:
    openssl req -x509 -newkey rsa:2048 -keyout private_key.pem -out public_cert.pem -days 365
openssl rsa -in private_key.pem -pubout -out public_key.pem
openssl pkcs8 -topk8 -inform PEM -outform PEM -nocrypt -in private_key.pem -out private_key_pkcs8.pem
  2. After generating the certificate, fill in the value of public_cert.pem in Workday.
    cat public_cert.pem

After filling in the information, click the OK button.

After completing the above steps, you will have an API Client named Tianle_Client for integration tasks.

Report

Overview

Report is a core function in the Workday platform, providing powerful data processing and report generation capabilities. Through Report, users can integrate scattered data into valuable data sets to provide data support for decision-making. It not only supports basic data extraction functions but can also handle complex business scenarios, including but not limited to:

  • Custom report design: Supports flexible field selection
  • Multi-data source integration: Can extract and associate data from multiple data sources
  • Advanced calculation functions: Supports complex data calculations and statistical analysis (Calculated Fields)
  • Dynamic parameter configuration: Allows users to set dynamic filtering conditions
  • Formatted output: Supports multiple output formats, such as Excel, PDF, etc.
  • Web Service integration: Can serve as a data service for other systems to call

As an indispensable data management tool in the Workday platform, Report provides comprehensive data visualization and analysis capabilities for enterprises.
Report is mainly applied in the following business scenarios:

  1. Data analysis and report generation
  2. System integration data source
  3. Business monitoring and decision support
  4. Compliance report generation

Creation Steps

  1. Use the following Task in Workday to create a Report:
    Create Custom Report
  2. Fill in basic information:
    • Report Name: Follow naming conventions
    • Report Type: Select Advanced
    • Enable Web Service: Check
    • Optimize Performance: Check

After filling in the information, click the OK button.

After completing the above steps, you will have a Report named Tianle_Test_Report for integration tasks.

Notes

  1. When selecting data sources, if business characteristics require real-time data, it is recommended to use non-indexed data sources.
  2. If you want to select non-indexed data sources, you need to uncheck the Optimized for Performance option.

Report as a Service (RaaS) Integration

Overview

Report as a Service (RaaS) is an important integration service provided by Workday that allows accessing report data through API calls.

Prerequisites

  1. Created Integration System User (ISU_Tianle)
  2. Configured API Client (Tianle_Client)
  3. Created custom report and enabled Web Service functionality (Tianle_Test_Report)
  4. Created Security Group required for integration tasks (Tianle_ISSG)

Using API to Call Report

  1. Use script to get JWT signature

Scripts have been written to generate signed JWT, not shown here.

# Since the JWT signing process needs to use the private key, this process cannot be completed directly in curl. You need to first generate a signed JWT through other tools (such as OpenSSL or dedicated JWT libraries), and then use it in the curl command.
JWT="eyJhbG...signed JWT token"
  1. Get AccessToken
# Then use JWT to get access token
curl -X POST "https://wd3-impl-services1.workday.com/ccx/oauth2/thoughtworks3/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer" \
  --data-urlencode "assertion=$JWT"
  1. Call API
ACCESS_TOKEN="Bearer eyJhbG...obtained AccessToken"
curl --request GET \
  --url 'https://wd3-impl-services1.workday.com/ccx/service/customreport2/thoughtworks3/ISU_Tianle/Tianle_Test_Report?format=json' \

Web Service Integration

Concept

Web Service integration is a standardized interface service provided by Workday that allows external systems to interact with data in Workday through HTTP/HTTPS protocol. Workday supports multiple Web Service types, including:

  1. REST Web Services

    • Based on REST architectural style
    • Uses JSON data format
    • Supports standard HTTP methods (GET, POST, PUT, DELETE, etc.)

  2. SOAP Web Services

    • Based on SOAP protocol
    • Uses XML data format
    • Provides WSDL interface description

The following paragraph is from Workday official documentation:

Workday REST APIs exist alongside our public SOAP-based web services. For transferring large sets of data, use the SOAP web services. SOAP web services best support batch transactions for external enterprise applications.

As mentioned in Workday official documentation, REST API coexists with SOAP API, and for batch data transfer, it is recommended to use SOAP API. Moreover, in Workday, SOAP API provides richer resource access, so in this article, the main example is SOAP API.

Note: SOAP API might not be common, if interested, you can refer to this document for a basic understanding: https://blog.postman.com/soap-api-definition/, not elaborated here.

Prerequisites

  1. Created Integration System User (ISU_Tianle)
  2. Configured API Client (Tianle_Client)
  3. Created Security Group required for integration tasks (Tianle_ISSG)

Authentication Methods

Workday Web Service supports the following authentication methods:

  1. Basic Authentication

    • Uses username and password
    • Suitable for simple integration scenarios
    • Relatively lower security

  2. OAuth 2.0

    • Supports JWT authorization flow
    • Provides higher security
    • Supports access token refresh

Based on our previous introduction to API Client, we choose OAuth 2.0 authentication method.

Summary and Reflections

Integration Key Points Summary

Through learning and practicing Workday integration, the following key points are worth noting:

  1. Permission Management is Crucial

    • Security Group is the core of permission management
    • Follow the principle of least privilege
    • Permission changes need to be activated promptly

  2. Choice of Authentication Method

    • OAuth 2.0 + JWT is the recommended authentication method
    • Need to properly store certificates and key information
    • Regularly update authentication information

  3. Choice of Integration Method

    • Choose appropriate integration method based on business scenarios
    • RaaS is suitable for scenarios requiring custom data queries
    • Web Service is suitable for complex business operations

Mastering the two integration methods mentioned in this article is sufficient to solve some integration problems. We hope this article can help colleagues who also need to perform Workday integration. If you have any questions, feel free to discuss.

Practical Experience

During the exploration process, we summarized the following practical experiences to share:

  • Due to the lack of clear official learning paths and documentation guidance, the learning curve is relatively steep. During the exploration process, significant time and effort need to be invested in trial and error, and verification through practical operations should be done whenever possible.
  • Workday official documentation content is sometimes insufficient, it is recommended to make good use of AI tools for assistance when searching for information, which can improve the efficiency of knowledge integration.
  • The Workday Community platform is recommended. Although the content is relatively brief, it contains a lot of practical experience sharing and is a good reference material.
  • Workday integration tasks usually require multiple steps working together to complete, so patience and attention to detail are needed during the operation.

Future Outlook

As a feature-rich and large system, truly mastering Workday’s integration capabilities is no easy task. Although self-learning and practice can solve some basic problems, systematic training and course learning are definitely essential to deeply understand this field.
